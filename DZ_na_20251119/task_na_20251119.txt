
Ответы на вопросы:

Теоретические вопросы (понимание)
Объясните разницу:
Какая разница между командой git add . и git add <имя_файла>? В каком случае вы бы использовали каждую из них?

Ответ:
Команда "git add ." используется для подготовки всех новых и изменённых файлов к коммиту, который будет выполнен после этой подготовки. То есть, если в конце команды "git add" указать точку, то будут обрабатываться именно все новые и измененные файлы.

Команда "git add <имя_файла>" используется для подготовки одного определенного файла, имя которого и указывается вместо "<имя_файла>".

То есть, команда "git add ." (с точкой) используется в тех случаях, когда нужно подготовить именно все измененные и новые файлы для отправки в хранилище, например, в GitHub. А команда "git add <имя_файла>" используется тогда, когда подготовить нужно только один определенный файл, имя которого указывается вместо "<имя_файла>".

Анатомия коммита:
Что такое коммит? Почему сообщения к коммитам должны быть краткими и информативными?

Ответ:
Коммит (commit) - один из вариантов перевода этого слова с английского означает "фиксировать". В git это используется для того, чтобы после завершения этапа работы зафиксировать изменения, внесенные в код. Для того, чтобы позже вспомнить, какие изменения были внесены, пишутся сообщения к коммитам.

Аналогия с упаковкой посылки. 
Если представить, что готовится посылка для отправки, то выполняются следующие действия:
- Рабочая директория — это стол, где разбросаны разные вещи (новые файлы и старые файлы, в которые были внесены изменения).
- "git add ." или "git add <имя_файла>" — это процесс, когда с этого стола собираются все эти вещи (новые файлы и старые файлы, в которые были внесены изменения) и складываются в коробку для посылки.
- "git commit" — это когда коробку заклеивают, подписывают её и отправляют (новые файлы и старые файлы, в которые были внесены изменения, сохраняются в истории).

Если проводить аналогию с игрой, то:
- проект — это игра,
- коммит — это сохранение игры, причем в этой "игре" есть возможность вернуться к любому из выполненных сохранений.

Как это работает:
- поработали над проектом (написали код, исправили ошибки, добавили элементы, сделали еще что-то),
- "сохранили игру" — после этих изменений сделали коммит,
- Git "фотографирует" все, что есть в проекте, и сохраняет эту "фотографию",
- можем спокойно продолжать работать, так как знаем, что в любой момент можем вернуться к этому сохранению позже.

Сообщения к коммитам должны быть краткими и информативными по следующим причинам:
- В "git log" видна только первая строка. То есть, чем короче и информативней запись, тем самому же автору внесенных изменений будет легче понять, что это за "фотография" проекта.
- Экономия времени - чем быстрее читается и понимается суть сообщения, тем быстрее можно продолжить работу над проектом.
- Хорошее сообщение отвечает на вопросы:
-- Что было изменено?
-- Почему это было изменено? Причем, контекст важнее деталей, потому что понимание контекста, облегчает понимание того, что можно делать с этой "фотографией" дальше. 
-- На что обратить внимание?
То есть, сообщение к коммиту должно позволить понять суть изменений, не заглядывая в сам код. От этого выигрывают все: 
- автор внесенных изменений, когда через полгода будет искать, когда были внесены определенные изменения,
- коллеги по работе, по той же причине, что облегчит им работу над этим кодом. 
Таким образом хорошие сообщения к коммитам могут стать чем-то вроде машины времени.
